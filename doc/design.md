# aigcpass 设计思想与实现细节

本文档记录 aigcpass 从传统命令行工具演化为 Claude Code Skill 过程中的核心设计决策、关键算法与工程权衡。

---

## 1. 项目定位：为什么做成 Skill

aigcpass 处理的是一个**半自动化、高审校依赖**的 workflow：

- 机器能做的事：解析 HTML、插入标记、调用 API、计算 diff、生成报告。
- 人必须做的事：判断片段是否跨段落、审阅降重文本的口语化问题、判定 API 适配是否忠实。

传统 CLI 工具的问题在于：每个阶段之间需要用户手动切换上下文（跑脚本 → 读输出 → 做决定 → 跑下一个脚本）。Claude Code Skill 的优势是 Agent 可以**编排整个流程**，只在需要人类判断的节点停下来。因此项目的核心设计目标不是"自动化一切"，而是"把机器擅长的部分串起来，把需要人类把关的节点暴露得清晰有序"。

---

## 2. 架构哲学：编排者 vs 执行者

Skill 中最重要的一个决策是**区分 Agent 跑的命令和用户跑的命令**。

| 类型 | 执行者 | 原因 |
|------|--------|------|
| 批处理脚本 | Agent | 无交互、可后台运行、输出可解析 |
| 交互式向导 | 用户 | 需要终端输入（`rich.prompt`） |
| 实时面板 | 用户 | 需要 `os.execv` 接管终端显示 TUI |
| `icdiff` 审阅 | 用户 | 需要 `less` 交互浏览 |

这个分界不是随意的，而是基于**终端控制权的归属**：如果一个命令会调用 `os.execv` 替换当前进程、或依赖 `curses`/`rich.Live` 实时刷新、或需要用户键盘输入，Agent 无法在不破坏自身会话状态的前提下执行它。SKILL.md 中的"谁会跑什么命令"表格是整个流程得以稳定运行的契约。

## 3. 两阶段流水线设计

### 3.1 Stage 1：提取与标记

**输入**：HTML 检测报告 + `main.tex`
**输出**：带 `% AIGC_BEGIN_{N}` / `% AIGC_END_{N}` 标记的 `main.tex` + CSV + 降重输入文件

#### 3.1.1 HTML 解析策略

检测报告中的 AIGC 片段以 `<a class="cl3">` 包裹。关键观察是：**相邻的 `cl3` 节点如果中间只有空白/标签，则属于同一个逻辑片段**。因此解析器先收集所有 `cl3` 节点，然后按"中间是否有实质文本"进行分组（`groups`），每组生成一个 `aigc_fragments` 元素。

#### 3.1.2 文本匹配算法

匹配的目标是将 HTML 中提取的纯文本片段定位到 `main.tex` 中的字节位置。挑战在于：

- LaTeX 源码中有大量命令（`\cite`, `\ref`, `\textbf` 等）在 HTML 中不存在。
- 同一个文本可能因排版断行出现在不同位置。

解决方案是**双向 strip 后匹配**：

1. 对 HTML 片段：去除所有空白，得到 `ns`。
2. 对 `main.tex` 的每一行：执行 `strip_for_match()`——删除注释、命令、环境、引用、格式化标记、特殊符号，再去除空白，得到 `stripped_lines`。
3. 将 `stripped_lines` 拼接为 `full_stripped`，在其中搜索 `ns`。
4. 如果完整匹配失败，回退到前 40/30/20/15/10 个字符的渐进式前缀匹配。

#### 3.1.3 段落边界扩展

找到片段位置后，不能只在片段起止处插入标记——因为片段可能只是段落的一部分，而 Stage 2 的 API 需要处理整个段落以保持上下文。因此算法**向上下扩展到空行边界**，以完整段落为单位包裹标记。

扩展后段落通过 `clean_to_plain()` 转换为纯文本，用于生成降重输入。该函数是 Stage 1 中最复杂的部分，本质上是一个**轻量级 LaTeX 到纯文本的反编译器**。

#### 3.1.4 `clean_to_plain()` 的设计

该函数不是通用 LaTeX 解析器，而是**面向中国高校论文模板的特化实现**。处理顺序经过精心安排：

1. **浮动体先行**：`figure`, `table`, `longtable`, `algorithm` 整体替换为 `<图:标题>` 或 `<表:标题>`。
2. **公式环境**：`equation`, `align` 替换为 `<公式>`。
3. **列表环境**：`enumerate`, `itemize` 提取 `\item` 内容，用 `; ` 连接。
4. **引用解析**：`\ref{key}` 通过预计算的 `ref_map` 替换为可读编号（见 3.1.5）。
5. **格式化命令剥除**：`\textbf{...}` → `...`，`\cite{...}` → `[...]` 等。
6. **行内公式**：`$...$`, `\(...\)` 保留公式内容。
7. **通用命令清除**：所有未被前面规则匹配的 `\command`。
8. **花括号展开**：`{...}` → `...`。
9. **特殊符号转义还原**：`\%` → `%`, `\&` → `&` 等。

处理顺序不可随意调换。例如，必须先处理浮动体再处理通用命令，否则 `\begin{figure}` 中的 `\includegraphics` 会被当成独立命令清除，导致残留垃圾文本。

#### 3.1.5 `build_ref_map()`：LaTeX 计数器模拟

`\ref{key}` 在编译后的 PDF 中显示为编号（如"3.2"），但源码中只有 `\label{fig:abc}` 和 `\ref{fig:abc}`。为了将 `\ref` 解析为可读的纯文本编号，必须模拟 LaTeX 的计数器机制。

核心逻辑：

- 扫描 `main.tex` 行 by 行。
- 遇到 `\section{...}`（非星号版本）时：`sec_num += 1`，重置 `fig_counter = 0`, `tab_counter = 0`。
- 遇到 `\begin{figure}` 时：标记 `in_figure = True`，清空 `pending_labels`。
- 遇到 `\end{figure}` 时：`fig_counter += 1`，为所有 `pending_labels` 生成编号 `sec_num.fig_counter`。
- `\label{...}` 收集规则：如果在浮动体内，加入 `pending_labels`；否则直接映射到当前 `sec_num`。

**关键决策**：`ref_map` 存储的是纯数字（如 `"3.2"`），而不是 `"图3.2"`。因为 LaTeX 源码中 `ef` 周围通常已经写了"图"字（如 `如图~\ref{fig:abc} 所示`），如果 `ref_map` 返回 `"图3.2"`，结果会变成`"如图 图3.2 所示"`。

#### 3.1.6 `MANUAL_POSITIONS` 回退机制

自动匹配基于文本内容，但当 HTML 中的片段因排版差异与 LaTeX 源码差异过大时（如英文摘要中的大小写/标点差异），匹配会失败。此时脚本输出 `[MISS]` 并提示用户将手动验证后的坐标填入 `MANUAL_POSITIONS` 字典。

坐标格式为 `(start_line, start_col, end_line, end_col)`，脚本通过 `line_col_to_byte()` 将其转换为字节偏移量。该函数反向模拟了 `strip_for_match()` 的可见字符计数逻辑，处理转义序列和花括号嵌套。

### 3.2 Stage 2：API 适配

**输入**：带标记的 `main.tex` + 降重后的 `AIGC片段优化.txt` + API 配置
**输出**：`疑似AIGC片段_待确认.csv`（含原片段和修改后片段）

#### 3.2.1 并发架构

Stage 2 的核心是**逐片段独立适配**，天然适合并发。使用 `ThreadPoolExecutor` 控制并发度（默认 3，可配置）。每个片段是一个独立的 API 调用，互不依赖。

**为什么不用 `asyncio`**：项目早期使用 `urllib.request` 进行 HTTP 调用，而 `urllib` 是同步 API。在同步 IO 场景下，`ThreadPoolExecutor` 是最简单的并发方案，且与 `rich.Live` 的线程安全模型兼容良好。

#### 3.2.2 实时面板

`rich.Live` 每 0.3 秒刷新一次表格，显示所有片段的状态：

| 列 | 含义 |
|---|---|
| 纯文本 diff rate | 用户降重修改了多少（`purple_text` vs `optimized_text`） |
| 适配 diff rate | API 实际修改了多少（`purple_text` vs `stripped(adapted_latex)`） |

这两个比率是**审阅阶段的先验数据**。纯文本 diff rate 告诉 Agent"这个片段本该有多大变化"；适配 diff rate 告诉 Agent"API 实际做了多大变化"。如果后者远小于前者，说明 API 没有忠实执行降重。

#### 3.2.3 验证体系

每次 API 返回后，`validate()` 执行以下检查：

1. **段落数匹配**：`count_units()` 按 `\n\n` 分割，确保原片段和适配后的段落单位数一致。
2. **LaTeX 环境完整性**：比较 `\begin{...}` 的集合，确保没有环境被意外删除或新增。
3. **引用保留**：`\cite{key}` 和 `\ref{key}` 的 key 集合必须完全一致。
4. **内容修改率**：`adapt_diff < 0.05` 视为 API 原样返回，判定失败。

验证失败会触发重试：温度递增（`base_temp + (attempt-1) * delta`），并将错误信息反馈给 LLM（`## 上一次输出验证失败`）。

#### 3.2.4 污染检查与恢复机制

Stage 2 会备份 `main.tex` 到 `result/stage2/main.tex.bak`。但如果用户在 Stage 2 中途修改了 `main.tex`（例如手动编辑了某个片段），再次运行 Stage 2 时备份就已经"污染"了——它不再代表 Stage 2 开始前的正确状态。

解决方案：**字节级比对**。如果 `bak` 存在且内容与当前 `main.tex` 不同，脚本拒绝运行并提示用户恢复。

```python
if os.path.exists(bak):
    if f.read() != tex:
        console.print("[red]ERROR:[/] bak 已被污染")
        sys.exit(1)
```

恢复命令由 SKILL.md 提供：
- 回 Stage 1 前：`cp jobs/{jobid}/result/stage1/main.tex.bak jobs/{jobid}/main.tex`
- 回 Stage 2 前：`cp jobs/{jobid}/result/stage2/main.tex.bak jobs/{jobid}/main.tex`

#### 3.2.5 断点续跑

`stage2_api.py` 支持 `--start` / `--end` 参数对指定范围重跑。同时，CSV 输出文件会被读取为 `existing` 字典——已经通过验证的片段会被跳过（`resume` 逻辑）。

**关键修复**：早期实现中，`--start 3 --end 3` 会覆盖整个 CSV 只保留第 3 行。修复方案是在写 CSV 时保留 `existing` 中不在当前 `jobs` 列表里的记录。

#### 3.2.6 "<不用改>" 机制

如果降重后的某段文本与原文几乎相同，用户可以在 `AIGC片段优化.txt` 中写入 `<不用改>`。脚本识别后会将该片段标记为 `skipped`，直接复制原片段到修改后片段列，节省一次 API 调用。

## 4. 三阶段审查体系

项目的核心创新之一是在"机器执行"之间插入了三个人类审阅节点，每个节点有明确的审查对象、判定标准和阻塞条件。

### 4.1 Stage 1 后：片段完整性审查

**审查对象**：HTML 中提取的紫色片段在插入标记前是否跨越了段落边界。

**为什么必须在此审查**：如果标记跨段落，Stage 2 的 API 会把两个无关段落当作一个整体处理，导致非 AIGC 内容被意外修改。

**判定原则**：
- 段落边界是硬约束（空行 `\n\n` = 段落分隔）。
- 头尾截断可接受（AIGC 检测系统本身就会标记句子片段）。
- 非紫色渗入（节标题、图表标题混进片段）是严重问题。

### 4.2 Stage 2 前：降重文本审查

**审查对象**：用户从降重工具拿回的 `AIGC片段优化.txt` 中是否存在口语化、文学化、非学术表达。

**为什么必须在此审查**：如果降重文本本身质量差，Stage 2 的 API 会忠实地将这些糟糕的措辞适配回 LaTeX——结果是"结构正确但内容不合格"的论文。

**六类检查项的设计来源**：
- "十分xxx"、形容词堆叠、顿号近义词：降重工具（尤其是基于同义词替换的工具）的常见输出模式。
- "吧"、文学化词语：LLM 降重（如 ChatGPT/DeepSeek）容易带入的口语/议论风格。
- 小括号补充：LLM 喜欢在括号里加解释性插入语，这在论文中属于非正式表达。

### 4.3 Stage 2 后：适配质量审查

**审查对象**：API 适配后的 LaTeX 是否忠实反映了用户降重的文本变化。

**核心指标：交集覆盖率**

```
Diff A = 紫色原文 vs 优化文本 的字符级差异幅度
Diff B = 紫色原文 vs 适配后文本（去 LaTeX 命令）的字符级差异幅度
coverage = Diff B / Diff A
```

**设计 rationale**：
- 我们不直接比较"优化文本"和"适配后文本"，因为两者格式不同（一个是纯文本，一个是 LaTeX）。
- 我们以"紫色原文"为基准，分别计算用户降重带来的变化（Diff A）和 API 适配带来的变化（Diff B）。
- 如果 API 忠实执行了降重，Diff B 应该与 Diff A 高度重叠（coverage ≥ 0.6）。

**为什么不是自动化判定**：覆盖率是参考，不是圣旨。某些片段可能覆盖率偏低但阅读后质量可接受；某些片段可能覆盖率合格但破坏了 LaTeX 结构（如 `\cite` 参数被修改）。因此 SKILL.md 明确要求"审查是 Agent 的定性判断，脚本只提供数据"。

## 5. 提示词工程：`stage2_fitback.md`

提示词模板采用**结构化输入 + 强约束列表 + 单一输出要求**的三段式设计：

1. **角色设定**："中文论文降重适配专家"——给 LLM 一个明确的领域身份。
2. **任务说明**：清晰定义输入（原片段、原文、优化后）和输出（只输出 LaTeX 源码）。
3. **关键要求加粗强调**：`"你必须进行措辞替换，不能原封不动地返回原片段。"`——这是针对早期实验中 API 直接回传原文本的对抗性设计。
4. **LaTeX 结构保留清单**：逐条列出不可修改的元素（引用、公式、环境、节标题等）。
5. **段落结构约束**：明确要求段落数和空行位置必须一致——这是 `validate()` 中段落数检查的前置条件。
6. **输出格式约束**：`"不要任何解释说明。不要用代码块包裹。"`——减少后处理清洗的复杂度。

## 6. 工程细节

### 6.1 路径解析：`script/_root.py`

项目从单一脚本演进为 Skill 后，文件层级关系发生了变化。早期使用固定层数的 `os.path.dirname()`，在 Skill 化后路径深度改变导致脚本找不到模板和 prompt。后来改用向上查找 `.git` 目录的方式，但这在 skill 被直接复制（非 git clone）或嵌套在其他 git 仓库中时会产生错误定位。

解决方案：统一使用 `script/_root.py` 提供 `ROOT` 变量。它基于 `_root.py` 自身的文件位置直接计算 skill 根目录（`_root.py` 位于 `script/` 下，所以它的上级目录就是 skill 根）。所有脚本导入 `from _root import ROOT` 即可，无需重复实现路径查找逻辑。这完全不依赖 git 仓库结构，无论 skill 被安装到文件系统的哪个位置都能正确定位。

### 6.2 配置隔离：per-job `api.yaml`

早期设计使用全局 `config/api.yaml`，但这样多个论文项目共用同一套 API key 和模型参数，既不方便也不安全。改为每 job 独立配置后：

- 不同 job 可以使用不同模型（如草稿用 `flash`，终稿用 `pro`）。
- API key 不会因为一个 job 的误操作影响其他 job。
- job 目录可以整体打包迁移或备份。

### 6.3 统一 CLI：`aigcpass.py`

虽然 Skill 模式下 Agent 直接调用 `script/` 下的脚本，但纯手动使用时需要一个统一入口。`aigcpass.py` 的设计原则是：

- `stage2` 子命令使用 `os.execv` 直通 `stage2_api.py`，确保实时面板能获得终端控制权。
- `diff` 子命令负责将 CSV 的两列提取为临时文件，调用 `icdiff` + `less` 进行分屏审阅。
- 其他命令透传参数给对应脚本。

### 6.4 CSV 编码

所有 CSV 文件使用 `utf-8-sig` 编码（带 BOM），确保 Excel 打开时中文不乱码。这是为中国高校用户场景做的兼容性决策。

## 7. 已知局限与权衡

| 局限 | 原因 | 缓解措施 |
|------|------|---------|
| 仅支持单文件 `main.tex` | 标记插入依赖全局字节偏移，多文件难以保证一致性 | README 明确告知用户 |
| 仅测试 Mac/Linux | 开发环境限制，Windows 路径和 TUI 行为差异大 | 文档声明 |
| 部分片段 API 始终 echo 原文 | DeepSeek 等模型对"修改 LaTeX 但不改结构"的任务理解不稳定 | 重试机制 + 手动修复 + `<不用改>` 机制 |
| `clean_to_plain()` 不是通用 LaTeX 解析器 | 完整解析器过于沉重，项目只需要覆盖论文常见命令 | 按需扩展命令列表 |
| 降重质量完全依赖外部工具 | 项目只负责"搬运"和"适配"，不改写语义 | 三阶段审查体系拦截劣质降重 |

## 8. 扩展性设计

### 8.1 自定义审查规则

`doc/check_optimization.md` 采用约定俗成的 Markdown 格式：

```markdown
### N. 检查项名称
**模式**：描述匹配规则
**示例**：| 不规范 | 建议修改 |
**检查方法**：`grep` 命令
```

Agent 读取该文件后按相同格式解析并执行。用户新增检查项只需遵循该格式，无需修改代码。

### 8.2 模型切换

`api.yaml` 支持 `provider: openai` 或 `anthropic`，`model` 字段可任意指定。`stage2_api.py` 的 `call_api()` 函数根据 provider 构造不同的请求体（OpenAI 的 `messages` 格式 vs Anthropic 的 `system` 字段）。项目开发过程中曾因 Anthropic endpoint 队列延迟过高而切换到 OpenAI-compatible endpoint，该抽象使得切换成本极低。

### 8.3 新命令添加

`aigcpass.py` 使用 `argparse` 子命令模式。新增子命令只需：
1. 在 `script/` 下实现脚本。
2. 在 `aigcpass.py` 中增加 `cmd_xxx()` 和 `sub.add_parser(...)`。
3. 在 `doc/scripts.md` 中补充参数说明。

## 9. 总结

aigcpass 的设计可以概括为一句话：**"机器做它能做好的，人在该把关的地方把关，Agent 负责把两者串成不漏气的流水线。"**

关键设计决策的优先级：

1. **正确性 > 自动化率**：三阶段审查宁可多停几次，也不让错误流入下游。
2. **可恢复 > 便利性**：多级备份 + 污染检查，确保任何时候都能回退到安全状态。
3. **可解释 > 性能**：diff 覆盖率、双比率面板、icdiff 逐段对比，所有中间产物对人类透明。
