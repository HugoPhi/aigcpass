# 降重文本审查：不规范用语检查

用户将降重后的 `AIGC片段优化.txt` 放入后，Stage 2 适配前，**你必须逐段审查降重文本**。

降重工具常引入论文不规范用语（口语化表达、冗余修饰、非学术语气）。你的任务是找到它们，生成报告，告知用户问题所在并提供修改建议。

审查是**定性判断**——不依赖脚本计算比率。你需要逐段阅读文本，用语言学判断决定是否违规。

---

## 检查清单

对 `AIGC片段优化.txt` 中每段文本，逐条检查以下 6 类问题：

### 1. "十分xxx"类程度副词滥用

**模式**：`十分` + 形容词/动词短语

论文中应使用更客观精确的表达，而非口语化的程度副词。允许出现的例外：`十分必要`、`十分重要`（固定搭配，但也不推荐）。

**示例**：
| 不规范 | 建议修改 |
|--------|---------|
| 十分清楚地展示了 | 清晰展示了 |
| 十分复杂 | 较为复杂 / 极为复杂 |
| 使用起来十分方便 | 使用方便 |

**检查方法**：`grep -n '十分' ~/.claude/skills/aigcpass/jobs/{jobid}/report/AIGC片段优化.txt`

### 2. 堆叠的形容词/副词修饰

**模式**：两个及以上形容词/副词连用修饰同一中心语

中文论文讲究简洁，避免修饰词堆叠。

**示例**：
| 不规范 | 建议修改 |
|--------|---------|
| 非常关键重要的 | 关键（二选一） |
| 极其特别地 | 极其 / 特别（二选一） |
| 较为非常困难 | 较为困难 / 非常困难 |

### 3. 顿号连接的近义形容词

**模式**：`A、B` 其中 A 和 B 语义相近或重叠

顿号连接的两个形容词如果语义重复，保留一个即可。

**示例**：
| 不规范 | 建议修改 |
|--------|---------|
| 精确、准确的 | 精确的 |
| 高效、快速的 | 高效的 |
| 复杂、繁琐的 | 复杂的 |

### 4. 语气词"吧"

**模式**：句末或句中出现的"吧"

`吧` 是口语语气词，论文中不应出现。

### 5. "自然""妥帖""毋庸"类文学化词语

**模式**：`自然`（作副词表示"当然"时）、`妥帖`、`毋庸`、`不言而喻`、`众所周知` 等

这些词带有文学/议论色彩，不符合工程论文客观陈述的风格。

**示例**：
| 不规范 | 建议修改 |
|--------|---------|
| 自然会被 | 因此 / 从而 |
| 妥帖地处理 | 妥善处理 / 有效处理 |
| 毋庸赘言 | 删除（直接陈述结论） |
| 不言而喻 | 显然 / 直接陈述 |
| 众所周知 | 直接引用文献或删除 |

### 6. 小括号补充说明

**模式**：`（...）` 中插入的解释性文字

论文中小括号通常用于引用编号 `[1]`、公式编号 `(3.2)`、缩写说明 `(MPK)`。用于补充解释属于口语化表达。

允许的例外：
- 括号内是**具体例子**：`如 xxx（例如 yyy）` — 可接受
- 括号内是**英文缩写**：`（MPK）`、`（EPT）` — 可接受
- 括号内是**引用或公式编号** — 可接受

**示例**：
| 不规范 | 说明 |
|--------|------|
| 这个方案很有效（我们后面会详细分析） | 口语化插入，应删除或独立成句 |
| 使用 MPK（Memory Protection Keys） | 缩写说明，可接受 |
| 例如 Nginx（作为 Web 服务器） | 例子说明，可接受 |

## 审查流程

### Step 1: 读取优化文本

```bash
cat ~/.claude/skills/aigcpass/jobs/{jobid}/report/AIGC片段优化.txt
```

### Step 2: 逐段检查

每段单独检查。用 `grep` 定位问题，然后人工读取上下文判断是否违规。

```bash
grep -n '十分' ~/.claude/skills/aigcpass/jobs/{jobid}/report/AIGC片段优化.txt
grep -n '吧' ~/.claude/skills/aigcpass/jobs/{jobid}/report/AIGC片段优化.txt
grep -n '自然\|妥帖\|毋庸\|不言而喻\|众所周知' ~/.claude/skills/aigcpass/jobs/{jobid}/report/AIGC片段优化.txt
grep -n '（.*）' ~/.claude/skills/aigcpass/jobs/{jobid}/report/AIGC片段优化.txt  # 手动检查每个括号
```

### Step 3: 生成审查报告

报告格式：

```
降重文本审查报告

审查了 N 个片段，共发现 K 处不规范用语。

---

片段 [ID]：
  • 类型：十分类
    位置：第 X 行 "...十分清楚地展示了..."
    建议：改为 "清晰展示了"
  • 类型：括号补充
    位置：第 Y 行 "...（我们后面会详细分析）"
    建议：删除括号内容，或独立成句

---

总结：
  - 十分类: N 处
  - 形容词堆叠: N 处
  - 顿号近义词: N 处
  - 语气词'吧': N 处
  - 文学化词语: N 处
  - 括号补充: N 处

建议：请修改上述问题后再进行 Stage 2 适配。
```

### Step 4: 报告用户

将报告展示给用户，**停下来**等用户修改。

用户确认后，继续 Stage 2。

---

## 判定原则

- **宁可多报也不少报**：不确定时，列出给用户自行判断
- **上下文优先**：某些词在特定语境下是可以接受的（如括号内是技术缩写）
- **不阻塞流程**：审查通过后立即进入 Stage 2，不要让用户等待
